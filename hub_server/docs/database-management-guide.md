# 数据库管理功能使用指南

## 概述

Hub Server 提供了完整的数据库管理功能，允许管理员通过 Web 界面监控、优化和清理数据库。

## 功能特性

### 1. 数据库统计

实时显示数据库状态：
- **数据库大小**: 当前数据库文件大小（MB）
- **总记录数**: 所有表的记录总数
- **数据表数量**: 数据库中的表数量
- **各表详情**: 每个表的记录数和占用空间

### 2. 数据库优化

执行以下优化操作：
- **ANALYZE**: 更新查询优化器统计信息，提升查询性能
- **VACUUM**: 回收已删除数据占用的空间，减小数据库文件大小
- **索引优化**: 重建索引以提高查询效率

**建议频率**: 每月执行一次，或在大量删除数据后执行

**注意事项**:
- 优化过程可能需要几分钟时间
- 优化期间数据库性能可能暂时下降
- 建议在低峰期执行

### 3. 数据归档

自动删除旧数据以释放空间：

**配置选项**:
- **浏览记录保留**: 保留最近 N 个月的浏览记录（默认 6 个月）
- **下载记录保留**: 保留最近 N 年的下载记录（默认 1 年）
- **同步历史保留**: 保留最近 N 个月的同步历史（默认 3 个月）

**警告**: 
- 归档操作会永久删除数据，无法恢复
- 执行前会显示确认对话框
- 建议先备份数据库

## 使用方法

### 访问数据库管理

1. 以管理员身份登录 Hub Server
2. 进入"管理后台"页面
3. 点击"数据库管理"标签

### 查看统计信息

进入数据库管理标签后，系统会自动加载统计信息，显示：
- 数据库总体状态（大小、记录数、表数量）
- 各数据表的详细统计（记录数、占用空间）

### 执行优化

1. 在"数据库优化"区域，点击"立即优化"按钮
2. 确认优化操作
3. 等待优化完成（通常需要 1-5 分钟）
4. 查看优化后的统计信息

### 执行归档

1. 在"数据归档"区域，配置保留时间：
   - 调整浏览记录保留月数（1-24 个月）
   - 调整下载记录保留年数（1-5 年）
   - 调整同步历史保留月数（1-12 个月）

2. 点击"执行归档"按钮

3. 确认删除操作（会显示将要删除的数据范围）

4. 等待归档完成

5. 查看删除的记录数和更新后的统计信息

## API 接口

### 获取数据库统计

```
GET /api/admin/database/stats
Authorization: Bearer <token>
```

响应示例：
```json
{
  "code": 0,
  "data": {
    "size_mb": "260.5",
    "total_records": 15234,
    "tables": [
      {
        "name": "browse_records",
        "record_count": 12450,
        "size_mb": "180.2"
      },
      {
        "name": "download_records",
        "record_count": 2784,
        "size_mb": "80.3"
      }
    ]
  }
}
```

### 优化数据库

```
POST /api/admin/database/optimize
Authorization: Bearer <token>
```

响应示例：
```json
{
  "code": 0,
  "message": "数据库优化完成"
}
```

### 归档旧数据

```
POST /api/admin/database/archive
Authorization: Bearer <token>
Content-Type: application/json

{
  "browse_months": 6,
  "download_years": 1,
  "history_months": 3
}
```

响应示例：
```json
{
  "code": 0,
  "data": {
    "browse_deleted": 3450,
    "download_deleted": 892,
    "history_deleted": 156,
    "total_deleted": 4498
  }
}
```

## 性能建议

### 数据库大小参考

- **< 500 MB**: 性能优秀，无需特别优化
- **500 MB - 1 GB**: 性能良好，建议每月优化一次
- **1 GB - 2 GB**: 性能可接受，建议每周优化一次
- **> 2 GB**: 建议执行归档操作，删除旧数据

### 优化时机

建议在以下情况执行优化：
- 大量删除数据后
- 数据库性能明显下降
- 定期维护（每月一次）
- 数据库文件大小异常增长

### 归档策略

根据业务需求调整保留时间：
- **高频使用**: 保留更长时间（浏览 12 个月，下载 2 年）
- **存储受限**: 保留更短时间（浏览 3 个月，下载 6 个月）
- **合规要求**: 根据数据保留政策设置

## 故障排除

### 优化失败

如果优化失败，可能原因：
- 数据库文件被锁定（其他进程正在使用）
- 磁盘空间不足（VACUUM 需要临时空间）
- 数据库文件损坏

解决方法：
1. 停止所有客户端连接
2. 检查磁盘空间（至少需要数据库大小的 2 倍空间）
3. 如果问题持续，使用 SQLite 工具检查数据库完整性

### 归档失败

如果归档失败，可能原因：
- 数据库连接超时
- 数据量过大导致操作超时

解决方法：
1. 减少保留时间，分批删除
2. 在低峰期执行
3. 检查数据库日志

## 备份建议

在执行归档操作前，建议备份数据库：

```bash
# 停止 Hub Server
# 复制数据库文件
cp hub_server.db hub_server.db.backup

# 或使用 SQLite 备份命令
sqlite3 hub_server.db ".backup hub_server.db.backup"
```
