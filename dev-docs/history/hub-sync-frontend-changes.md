# Hub 前端同步页面修改说明

## 修改概述

由于系统改为纯 WebSocket 推送模式，客户端会自动定时推送数据到 Hub Server，因此前端不再需要"手动触发同步"功能。


## 保留的功能

### 1. 查看同步状态

用户仍然可以：
- 查看所有设备的同步状态
- 查看同步统计（总设备、同步中、成功、失败）
- 筛选和搜索设备
- 自动刷新（每 30 秒）

### 2. 查看详情

用户可以查看单个设备的：
- 设备信息
- 同步统计（浏览记录、下载记录）
- 最后同步时间
- 错误信息（如果有）

### 3. 查看历史

用户可以查看设备的同步历史记录：
- 同步时间
- 同步类型（浏览/下载）
- 同步数量
- 状态（成功/失败）
- 错误信息

## 用户体验改进

### 1. 更清晰的说明

页面顶部明确说明"客户端自动推送数据，无需手动触发"，避免用户困惑。

### 2. 简化操作

删除了不必要的"同步"按钮，界面更简洁，用户只需关注数据查看。

### 3. 自动刷新

保留了自动刷新功能（30 秒），用户可以实时看到最新的同步状态。

## 后端 API 修改

### TriggerSync 接口

虽然前端不再调用此接口，但后端保留了该接口，并修改了返回消息：

**修改前**:
```json
{
  "code": 0,
  "message": "Sync started"
}
```

**修改后**:
```json
{
  "code": 0,
  "message": "正在检查设备状态，客户端会自动推送数据"
}
```

这样即使有其他地方调用此接口，也能得到正确的提示。

## 数据流程

### 旧流程（HTTP 拉取）

```
用户点击"同步" → 前端调用 API → Hub Server 请求客户端 → 客户端返回数据
```

### 新流程（WebSocket 推送）

```
客户端定时器触发 → 客户端推送数据 → Hub Server 接收并保存 → 前端自动刷新显示
```

### 配置变更

客户端配置需要启用推送：

```yaml
hub_sync:
  enabled: true
  push_enabled: true
  push_interval: 5m
  push_batch_size: 1000
```

### 数据迁移

无需数据迁移，现有的同步状态和历史记录会继续保留。

## 常见问题

### Q: 为什么删除了"立即同步"按钮？

A: 因为现在使用 WebSocket 推送模式，客户端会自动定时推送数据，不需要手动触发。

### Q: 如果我想立即看到最新数据怎么办？

A: 点击"刷新"按钮即可查看最新的同步状态。客户端会按配置的间隔（默认 5 分钟）自动推送数据。

### Q: 同步失败了怎么办？

A: 系统会自动重试。客户端会在下次定时任务时继续推送，使用增量同步，不会丢失数据。

### Q: 可以调整推送频率吗？

A: 可以。在客户端的 `config.yaml` 中修改 `hub_sync.push_interval` 参数。

### Q: 前端还能看到同步历史吗？

A: 可以。点击"同步历史"按钮即可查看完整的同步历史记录。

## 总结

通过这些修改，前端界面更加简洁明了，用户体验更好。系统采用自动推送模式，无需用户手动干预，降低了使用门槛。
