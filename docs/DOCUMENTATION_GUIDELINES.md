# 文档规范指南

## 📋 概述

本文档定义了项目文档的编写规范和组织标准，确保文档的一致性和可维护性。

## 📁 目录结构规范

### 标准目录结构

```
项目根目录/
├── README.md                      # 项目主文档（必须）
│
├── docs/                          # 文档目录
│   ├── README.md                  # 文档总览（必须）
│   ├── INDEX.md                   # 文档索引（必须）
│   ├── 主要文档/                   # 用户和开发者文档
│   ├── guides/                    # 使用指南
│   └── fix/                       # 修复文档（私有）
│
└── scripts/                       # 工具脚本
    └── *.md                       # 工具文档
```

### 目录用途

| 目录 | 用途 | 公开性 | 文档类型 |
|------|------|--------|----------|
| `docs/` | 主要文档 | ✅ 公开 | 介绍、安装、功能、配置、API、帮助 |
| `docs/guides/` | 使用指南 | ✅ 公开 | 快速开始、访问说明、问题诊断 |
| `docs/fix/` | 修复文档 | ❌ 私有 | Bug修复、优化、诊断、测试 |
| `scripts/` | 工具文档 | ✅ 公开 | 工具说明、快速参考、索引 |

## 📝 文档命名规范

### 主要文档（docs/）

**格式**: `大写字母_下划线.md`

**示例**:
- ✅ `INSTALLATION.md`
- ✅ `CONFIGURATION.md`
- ✅ `API.md`
- ❌ `installation.md`
- ❌ `Install-Guide.md`

**规则**:
- 使用全大写字母
- 单词间用下划线连接
- 简洁明了，一般1-3个单词
- 使用英文命名

### 使用指南（docs/guides/）

**格式**: `描述性名称_类型.md`

**示例**:
- ✅ `BUILD_QUICK_START.md`
- ✅ `WEB_CONSOLE_ACCESS.md`
- ✅ `DECRYPTION_TROUBLESHOOTING.md`
- ❌ `quick-start.md`
- ❌ `WebConsole.md`

**规则**:
- 使用全大写字母
- 描述性名称 + 类型后缀
- 类型后缀: `_START`, `_ACCESS`, `_TROUBLESHOOTING`, `_GUIDE`

### 修复文档（docs/fix/）

**格式**: `描述性名称_类型.md` 或 `中文描述.md`

**示例**:
- ✅ `COMMENT_COUNT_FIX.md`
- ✅ `SCROLL_OPTIMIZATION.md`
- ✅ `日志系统完成总结.md`
- ✅ `Profile日志修复说明.md`

**类型后缀**:
- `_FIX.md` - 问题修复
- `_OPTIMIZATION.md` - 性能优化
- `_GUIDE.md` - 使用指南
- `_SUMMARY.md` - 总结文档
- `_TEST_GUIDE.md` - 测试指南
- `_SUCCESS.md` - 成功实现
- `_ISSUE.md` - 问题描述
- `DEBUG_*.md` - 调试文档

### 工具文档（scripts/）

**格式**: `大写字母_下划线.md`

**示例**:
- ✅ `README.md`
- ✅ `QUICK_REFERENCE.md`
- ✅ `INDEX.md`

## 📄 文档内容规范

### 文档结构

每个文档应包含以下部分：

```markdown
# 文档标题

## 概述
简短描述文档内容和用途（1-2段）

## 目录（可选，长文档需要）
- [章节1](#章节1)
- [章节2](#章节2)

## 主要内容
### 章节1
内容...

### 章节2
内容...

## 相关链接
- [相关文档1](link1.md)
- [相关文档2](link2.md)

## 更新日志（可选）
- 2025-11-23: 初始版本

---
**最后更新**: 2025-11-23
```

### 标题规范

**层级**:
```markdown
# 一级标题（文档标题，每个文档只有一个）
## 二级标题（主要章节）
### 三级标题（子章节）
#### 四级标题（细节，尽量少用）
```

**规则**:
- 标题要简洁明了
- 使用动词开头（如：安装、配置、使用）
- 避免过深的层级（最多4级）

### 内容规范

#### 1. 代码块

**格式**:
````markdown
```语言
代码内容
```
````

**示例**:
````markdown
```bash
go build -o wx_channel.exe
```

```javascript
console.log('Hello World');
```

```powershell
Get-ChildItem -Path docs
```
````

#### 2. 列表

**无序列表**:
```markdown
- 项目1
- 项目2
  - 子项目2.1
  - 子项目2.2
```

**有序列表**:
```markdown
1. 第一步
2. 第二步
3. 第三步
```

**任务列表**:
```markdown
- [x] 已完成任务
- [ ] 未完成任务
```

#### 3. 表格

**格式**:
```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容1 | 内容2 | 内容3 |
| 内容4 | 内容5 | 内容6 |
```

**对齐**:
```markdown
| 左对齐 | 居中 | 右对齐 |
|:-------|:----:|-------:|
| 内容   | 内容 | 内容   |
```

#### 4. 链接

**内部链接**:
```markdown
[文档标题](相对路径.md)
[章节标题](#章节标题)
```

**外部链接**:
```markdown
[链接文字](https://example.com)
```

#### 5. 图片

```markdown
![图片描述](图片路径)
```

#### 6. 引用

```markdown
> 这是引用内容
> 可以多行
```

#### 7. 强调

```markdown
**粗体**
*斜体*
~~删除线~~
`行内代码`
```

### Emoji 使用规范

**推荐使用的 Emoji**:

| Emoji | 用途 | 示例 |
|-------|------|------|
| 📋 | 概述、列表 | `## 📋 概述` |
| 📁 | 目录、文件 | `## 📁 目录结构` |
| 📚 | 文档、资料 | `## 📚 文档列表` |
| 📝 | 说明、注释 | `## 📝 使用说明` |
| 🎯 | 目标、重点 | `## 🎯 快速导航` |
| ✅ | 完成、成功 | `- ✅ 已完成` |
| ❌ | 错误、失败 | `- ❌ 不推荐` |
| ⚠️ | 警告、注意 | `⚠️ 注意事项` |
| 💡 | 提示、建议 | `## 💡 使用建议` |
| 🔧 | 工具、配置 | `## 🔧 配置选项` |
| 🚀 | 开始、启动 | `## 🚀 快速开始` |
| 🔍 | 搜索、查找 | `## 🔍 查找文档` |
| 🔗 | 链接、关联 | `## 🔗 相关链接` |
| 📊 | 统计、数据 | `## 📊 文档统计` |
| 🎉 | 庆祝、完成 | `🎉 整理完成！` |

**使用规则**:
- 适度使用，不要过多
- 保持一致性
- 主要用于标题和重点内容

## 🔄 文档维护规范

### 添加新文档

**步骤**:

1. **确定文档类型**
   - 主要文档 → `docs/`
   - 使用指南 → `docs/guides/`
   - 修复文档 → `docs/fix/`

2. **按规范命名**
   - 参考命名规范
   - 使用合适的后缀

3. **编写内容**
   - 遵循内容规范
   - 包含必要的章节

4. **更新索引**
   - 在 `docs/INDEX.md` 中添加链接
   - 在对应目录的 `README.md` 中添加说明

5. **检查链接**
   - 确保所有链接有效
   - 检查相对路径正确

### 更新现有文档

**步骤**:

1. **修改内容**
   - 保持格式一致
   - 更新相关章节

2. **更新日期**
   - 在文档末尾更新"最后更新"日期
   - 在"更新日志"中添加记录

3. **检查影响**
   - 检查是否影响其他文档
   - 更新相关链接

### 删除文档

**步骤**:

1. **确认删除**
   - 确认文档已过时或不再需要
   - 检查是否有其他文档引用

2. **更新索引**
   - 从 `docs/INDEX.md` 中移除
   - 从目录 `README.md` 中移除

3. **更新链接**
   - 修复所有指向该文档的链接
   - 或重定向到替代文档

### 归档文档

对于过时但有参考价值的文档：

1. 创建 `docs/archive/` 目录
2. 移动文档到归档目录
3. 在原位置添加重定向说明
4. 更新索引标记为"已归档"

- [ ] 内容更新完整
- [ ] 更新了"最后更新"日期

## 📊 文档质量标准

### 优秀文档的特征

1. **清晰性**
   - 结构清晰，层次分明
   - 语言简洁，易于理解
   - 重点突出，逻辑清楚

2. **完整性**
   - 包含必要的信息
   - 提供足够的示例
   - 有相关链接和参考

3. **准确性**
   - 信息准确无误
   - 代码可以运行
   - 链接都是有效的

4. **可维护性**
   - 格式规范统一
   - 易于更新修改
   - 有版本记录

5. **用户友好**
   - 考虑读者需求
   - 提供快速导航
   - 有实用的示例
